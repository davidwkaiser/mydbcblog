Rails: A Framework for Ruby

{Ruby => Rails; Python => Django}

March 9, 2016

So, as the little note above suggests, in a former life I was a linguist. As such, I would have seen the phrase "Ruby on Rails" and thought "cool, a colocation" (a colocation is a group words that often occur together), then moved on to some other topic. Academics can be easily distracted...

Right, back to Rails. So I would hear this phrase. As I started teaching myself Python, I learned about Frameworks, and how they can be used to both simplify and extend the functionality of a programming language. Ruby is a great language, and it can do lots of fun things, but it seems like most of the Ruby jobs mention website development using the Rails framework.

We all know Ruby's creation story: Matz wanted an object-oriented scripting language, and didn't like anything out there at the time, so he created Ruby. Cool. Thanks, Matz!

Rails was originally developed as part of the Basecamp (formerly 37 Signals) project management application, and it was later abstracted out and published on its own as a framework for web development for Ruby.

Rails' core philosophy is "Convention over Configuration," meaning that you don't need to actively specify standard items, they would have default settings which would be applied unless the developer consciously chooses to override them.

Rails uses the model-view-controller (MVC) configuration. A model is mapped to a (default) database. The controller responds to requests from surver-side applications and offers views in HTML. Rails philosophy here is "Fat models, skinny controllers," meaning that most of the logic should be baked into the model, not the controller.

Rails also plays nicely with JavaScript, using several JS libraries and JQuery.

Lastly, Rails is only one of many frameworks for Ruby. Others include Sinatra, Padrino and Lotus.

That said, the Django Pony is way cooler than the Rails logo.
